## Phase 1 — Backend Scaffolding (C++)

### Steps

1. Set up `CMakeLists.txt` with:
   - C++17 standard
   - Google Test as a dependency (via FetchContent)
   - Coverage flags (`--coverage` / `gcov`)
2. Implement `BigInteger` class:
   - Addition, subtraction, multiplication, modulo
   - Modular exponentiation (`mod_pow`)
   - Miller-Rabin primality test
3. Implement `RSA` class:
   - `generate_prime(bits)` — generate random prime
   - `generate_keypair(bits)` — returns `{n, e, d}`
   - `encrypt(m, e, n)` → ciphertext
   - `decrypt(c, d, n)` → plaintext
   - `sign(hash, d, n)` → signature
   - `verify(signature, e, n)` → hash
4. Expose a `/api/health` endpoint to confirm server runs

### Verification Checkpoint
- Unit tests pass for BigInteger and RSA with known small-number examples
- `/api/health` returns 200
- Code coverage ≥ 90% on math and RSA modules

### Steps

### Verification Checkpoint

## Phase 4 — Certificate Authority & Digital Certificate Module

### Steps

1. Create educational content endpoint `GET /api/education/certificate-theory`
   - What is a digital certificate (X.509-style explanation)
   - What fields it contains: subject, issuer, public key, signature, validity
   - Proof example: show a mock certificate with dummy values, explain each field
2. Implement `Certificate` class (backend):
   - Fields: subject, issuer, public_key (n, e), serial, validity_period, ca_signature
   - `CA::issue_certificate(subject_name, subject_public_key)` → Certificate
   - CA signs `hash(subject + public_key + serial)` with its private key
3. Create endpoint `POST /api/ca/issue-certificate`
   - Alice requests certificate, passing her public key
   - CA validates, signs, returns certificate
   - Response includes: packets exchanged log (request packet → CA signs → certificate packet)
4. Frontend: "CA Certificate" page
   - First: show certificate theory reading material
   - Then: simulate packet exchange animation (Alice → CA, CA → Alice)
   - Display issued certificate with all fields labeled

### Verification Checkpoint
- Certificate theory displayed before simulation
- Packet exchange log shows exactly 2 packets (request + certificate)
- Certificate fields all populated and CA signature is valid
- Unit tests cover certificate issuance and CA signature

## Phase 5 — Document Upload & Signing Module

### Steps

1. Create endpoint `POST /api/alice/upload-document`
   - Accepts a text file upload
   - Returns: file content, SHA-256 hash (custom implementation, no stdlib crypto)
2. Implement `Hash` class:
   - Custom SHA-256 from scratch (no `<openssl>` or similar)
3. Create endpoint `POST /api/alice/sign-document`
   - Alice signs the document hash using her RSA private key
   - Returns: original hash, signature (hex), certificate reference
4. Frontend: "Sign Document" page
   - File upload input
   - Display file content on upload
   - Show computed hash
   - Show generated signature
   - "Send to Bob →" button

### Verification Checkpoint
- Hash output matches known SHA-256 test vectors
- Signature is generated and displayed in hex
- UI shows all intermediate steps clearly
- 90% coverage on Hash and signing logic

## Phase 6 — Bob's Verification Module

### Steps

1. Create educational content endpoint `GET /api/education/certificate-verification`
   - How Bob verifies a certificate: checks CA signature using CA's public key
   - Step-by-step reading material
2. Create educational content endpoint `GET /api/education/signature-verification`
   - How Bob verifies Alice's signature: decrypt signature → compare hash
   - Step-by-step reading material
3. Create endpoint `POST /api/bob/verify`
   - Input: Alice's certificate + document + signature
   - Step 1: Verify certificate (CA signature check) → result
   - Step 2: Extract Alice's public key from certificate
   - Step 3: Compute hash of document
   - Step 4: Decrypt signature using Alice's public key
   - Step 5: Compare hashes → verified / not verified
   - Returns full step-by-step log with result
4. Frontend: "Bob Verifies" page
   - Show certificate verification theory → then run verification
   - Show signature verification theory → then run verification
   - Final banner: "✓ Verified: This document is from Alice"

### Verification Checkpoint
- Certificate verification steps displayed as reading before execution
- Signature verification steps displayed as reading before execution
- Successful verification banner shows correctly
- All verification steps logged and visible on UI

## Phase 7 — Attacker (Fake-Alice) Module

### Steps

1. Create endpoint `POST /api/fake-alice/attempt`
   - Fake-Alice has her own RSA key pair (no CA certificate)
   - She signs a (possibly modified) document with her own private key
   - Sends fake certificate (self-signed or forged) + document + signature to Bob
2. Extend `POST /api/bob/verify` to handle fake-alice scenario:
   - Certificate check fails: CA signature invalid
   - Signature check fails: hash mismatch
   - Returns failure log with specific failure point
3. Frontend: "Attacker Scenario" page
   - Walk through same steps as Phase 6 but with Fake-Alice's data
   - At failure point: red banner "✗ Verification Failed: Certificate not trusted"
   - Or if cert passes but sig fails: "✗ Signature Mismatch: Document tampered"
   - Educational callout explaining why each check failed

### Verification Checkpoint
- Attacker scenario always fails verification
- Failure reason is specific and educational
- UI shows red failure state clearly
- Unit tests cover both failure paths

## Phase 8 — Custom TLS Simulation Layer

### Steps

1. Design a simplified TLS 1.3-like handshake (no pre-built TLS libs):
   - ClientHello: Bob sends supported params
   - ServerHello: Alice responds with certificate + chosen params
   - Key Exchange: use RSA to exchange a session key
   - Finished: both sides verify handshake integrity with HMAC (custom)
2. Implement `TLSSession` class in backend
3. Create endpoint `POST /api/tls/handshake`
   - Simulates the handshake steps
   - Returns each step's packet content for display
4. Implement custom HMAC (no stdlib)
5. Frontend: "TLS Handshake" page
   - Packet-by-packet animation of handshake
   - Each packet labeled with its TLS message type

### Verification Checkpoint
- Full handshake completes successfully with Alice
- Handshake fails (or detects intrusion) when Fake-Alice injects
- HMAC verified at Finished step
- Unit tests cover handshake state machine

## Quality Standards

- C++ code follows Google C++ Style Guide
- All functions documented with Doxygen comments
- No use of `goto`, raw owning pointers (use smart pointers)
- Frontend: ESLint + Prettier enforced in CI
- No pre-built library for RSA, key exchange, SHA-256, or HMAC
- `cpp-httplib` or similar allowed only for HTTP routing, not crypto
- Every public function has at least one unit test
- Integration tests cover all happy paths and failure paths

---

## Notes for Students

Each phase in the UI is gated — students must read the educational material before
the simulation runs. This ensures concepts are understood before being demonstrated.
The attacker scenario (Phase 7) is intentionally placed after the successful verification
so students can contrast what a valid vs. invalid verification looks like.
